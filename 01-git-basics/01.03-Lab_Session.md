# Hands-On Lab: GitHub Authentication & Collaboration Workflow

## Lab Objectives
By the end of this session, you will be able to:
1. Configure **HTTPS** authentication using a Personal Access Token (PAT) & Git Credential Manager.
2. Generate and configure **SSH Keys** for secure, passwordless access (The DevOps Standard).
3. Manage remote repositories (`origin` and `upstream`).
4. Execute the Open Source contribution workflow (Fork → Branch → PR).

---

## Phase 1: Personal Workflow (HTTPS + PAT)
*Best for quick starts. We will use the Git Credential Manager so you don't have to type your PAT every time.*

### Step 1: Repository Creation & Credential Setup
1. Go to GitHub Web → **New Repository** (Name: `github-auth-lab`, Public, Initialize with README).
2. Open your terminal (Git Bash).

| Action | Command | Verification (How to know it worked) |
| :--- | :--- | :--- |
| **Enable Credential Manager** | `git config --global credential.helper manager` | No output. (Fails if you get an "invalid key" error). |
| **Clone Repo** | `git clone https://github.com/<USERNAME>/github-auth-lab.git`<br>`cd github-auth-lab` | Terminal shows `Resolving deltas: 100%`. You are now in the `github-auth-lab` folder. |

### Step 2: Push Changes (Triggering the PAT)
| Action | Command | Verification (How to know it worked) |
| :--- | :--- | :--- |
| **Make a Change** | `echo "HTTPS Lab" >> notes.txt` | `cat notes.txt` displays "HTTPS Lab". |
| **Commit** | `git add notes.txt`<br>`git commit -m "Add HTTPS notes"` | Terminal confirms `1 file changed`. |
| **Push** | `git push origin main` | **Prompt appears.** Enter your GitHub Username and paste your **PAT** as the password. Terminal shows `main -> main`. |
| **Test Cache** | Make another small change, commit, and run `git push origin main` again. | **Success:** Pushes immediately without asking for your PAT again. |

## Phase 2: Professional Setup (SSH Authentication)
*The industry standard for DevOps. This allows secure, passwordless authentication using cryptographic keys.*

### Step 3: Generate and Link SSH Keys
| Action | Command | Verification (How to know it worked) |
| :--- | :--- | :--- |
| **Generate Key** | `ssh-keygen -t ed25519 -C "your_email@example.com"` | Terminal prompts for a file location. **Press Enter 3 times** to accept defaults and skip the passphrase. |
| **View & Copy Public Key** | `cat ~/.ssh/id_ed25519.pub` | Terminal outputs a long string starting with `ssh-ed25519`. Copy this entire line. |

**Action on GitHub:** Go to **Settings** → **SSH and GPG keys** → **New SSH key**. Paste the copied key and save.

### Step 4: Test & Clone via SSH
| Action | Command | Verification (How to know it worked) |
| :--- | :--- | :--- |
| **Test Connection** | `ssh -T git@github.com` | Terminal asks `Are you sure...?` Type `yes`. Then outputs: *"Hi <username>! You've successfully authenticated."* |
| **Clone via SSH** | `cd ..`<br>`git clone git@github.com:<USERNAME>/github-auth-lab.git` | Clones successfully without asking for any passwords. |

## Phase 3: Team Collaboration (Forking & Remotes)
*Simulating a real-world team environment where you cannot push directly to the main code base.*

### Step 5: Fork and Configure Remotes
1. Go to the instructor's or a teammate's repository on GitHub and click **Fork**.

| Action | Command | Verification (How to know it worked) |
| :--- | :--- | :--- |
| **Clone Your Fork** | `git clone git@github.com:<USERNAME>/<FORKED-REPO>.git`<br>`cd <FORKED-REPO>` | You are inside the forked repo's directory. |
| **Add Upstream** | `git remote add upstream git@github.com:<ORIGINAL-OWNER>/<REPO>.git` | No output. |
| **Verify Remotes** | `git remote -v` | You should see 4 lines: two for `origin` (your fork) and two for `upstream` (the original repo). |

### Step 6: The Pull Request (PR) Workflow
| Action | Command | Verification (How to know it worked) |
| :--- | :--- | :--- |
| **Create Feature Branch** | `git checkout -b feature/auth-update` | Terminal outputs: `Switched to a new branch 'feature/auth-update'`. |
| **Make Changes & Commit** | `echo "PR test" >> update.md`<br>`git add update.md`<br>`git commit -m "Add PR test"` | Terminal confirms commit. |
| **Push to Origin** | `git push origin feature/auth-update` | Terminal provides a direct URL to GitHub to create your Pull Request. |

**Action on GitHub:** Click the URL from the terminal or go to your fork on GitHub and click **Compare & Pull Request**. Submit the PR to the original repository.

## 🔮 Looking Ahead: Introduction to Git Conflicts
In this lab, you successfully pushed changes and raised a Pull Request in a clean, straight line. But what happens in a real company when multiple developers are editing the *exact same file* at the *exact same time*?

**Enter: The Git Merge Conflict.**

Next session, we will cover:
* **Why Conflicts Happen:** When Git's automatic merging fails because it doesn't know which developer's code to keep.
* **How to Read a Conflict:** Understanding the `<<<<<<< HEAD` and `======` markers in your files.
* **Resolution Strategy:** How to fetch from `upstream`, rebase your branch, manually select the correct code, and force push your clean changes.
